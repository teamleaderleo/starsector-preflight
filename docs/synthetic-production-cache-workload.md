# Extended production-cache workload

This workload composes the deterministic extended provider index with the
production prepared-audio and generated-bytecode cache implementations. It
replaces the duplicate experimental stores left in PR #60 with the already
merged SPAU and SPJB formats.

## Prepared audio

The generated profile emits canonical RIFF/WAVE PCM files. The synthetic WAV
preparation path validates the exact 44-byte header, RIFF and data lengths,
PCM format, channel count, sample rate, byte rate, block alignment, and signed
16-bit little-endian layout before constructing `PreparedAudio`.

Its decoder-policy identity hashes the exact implementation class bytes plus an
explicit policy suffix. Effect files use `FULLY_DECODED_EFFECT` and may create
SPAU blobs. Music files use `STREAMED`, remain ineligible, and never carry PCM
or create cache files.

The tiny workload contains 10 effect files and 2 streamed files. The medium
workload contains 40 effect files and 10 streamed files.

## Generated bytecode

All generated Java source providers are compiled as one complete source set.
The production `GeneratedBytecodeCacheWrapper` receives an exact context that
includes:

- canonical source-set SHA-256;
- exact Java executable path, size, and SHA-256;
- compiler options;
- compiler seam class bytes and implementation suffix;
- stable defining-loader identity for that exact Java runtime; and
- exact profile and provider-index dependency snapshots.

The JDK compiler seam bounds source count, per-source and total source bytes,
per-class and total class bytes, and diagnostics. It returns every generated
top-level, nested, and anonymous class. The worker loads the returned map in a
fresh class loader and invokes the final generated source class, so a cache hit
must remain executable.

## Separate-JVM proof

The focused workflow runs tiny profiles on Linux, macOS, and Windows and a
2,516-file medium profile on Ubuntu. Each test performs:

1. a cold JVM that builds the provider index, prepares every eligible WAV, and
   compiles the complete Java source set;
2. a warm JVM with an exact provider-index hit, one SPAU hit per effect, zero
   WAV decoder calls, one SPJB hit, and zero compiler calls;
3. a pass after independently corrupting one real SPAU file and one real SPJB
   file, requiring exactly one audio decode and one complete-source compile;
4. exact profile, provider, audio, bytecode, context-key, and combined output
   digest comparisons across all passes.

Cache read or write failures remain diagnostic. Original source processing
continues, and returned original bytecode maps retain the production wrapper's
existing exception and mutability contracts.

This is a generated compatibility and reuse proof. It does not use
Starsector's shipped audio decoder or runtime compiler, does not represent a
real mod installation, and does not establish a startup-time change.
