# Creative exploration

Use this optional workflow when a consequential human-facing decision has several credible answers and the preferred direction is unclear. In Starsector Preflight, this applies mainly to command output, reports, operator prompts, documentation, comparison presentation, and asset-tool interfaces.

The sequence is:

1. **Diverge:** produce a few intentionally different candidates against the same brief and exact evidence.
2. **Converge:** make a human decision, choose one direction, and continue iterating there.

Parallel candidates help discover a presentation direction. A continuing implementation thread is usually better once the direction is known.

## Direction or execution?

- **Direction uncertainty:** several substantially different presentations could satisfy the requirements. Explore alternatives.
- **Execution uncertainty:** the intended result is clear and needs refinement. Stay on one branch and iterate.

Choosing the form of a comparison report, a texture-memory summary, an operator prompt sequence, or a font-generation report may justify alternatives. Correctness repairs, resource discovery, cache control, hashing, readiness detection, and validation remain verified implementation work.

## Prepare the brief

Separate fixed requirements from open decisions.

### Fixed

Every candidate should preserve:

- exact installation and resource identities;
- hashes, paths, manifests, and retained evidence;
- explicit permission boundaries;
- comparison randomization and per-half isolation;
- failure classifications and refusal behavior;
- licensing constraints;
- deterministic output needed by scripts;
- accessibility and supported terminal or browser behavior.

### Open

Name the questions the exploration should answer:

- Which result belongs first?
- What should an operator see before approving an action?
- Which evidence belongs inline versus in an appendix?
- How should uncertainty, invalid comparisons, and partial success appear?
- Which language makes a recovery step safe to follow?
- How should memory, texture, and font reports balance density with explanation?

## References and anti-references

Use a small number of relevant CLI tools, diagnostic reports, modding utilities, current Preflight output, and prior operator transcripts. Record the specific quality worth examining.

Also record patterns to avoid: vague green success banners, hidden cache mutation, destructive actions presented as routine cleanup, unlabeled memory estimates, generic progress spinners, or polished summaries that separate claims from retained evidence.

References provide vocabulary. Preflight’s observed files, hashes, manifests, and validated lifecycle evidence remain authoritative.

## Choose the amount of divergence

- **Small variation:** compare ordering, labels, grouping, prompts, tables, or disclosure.
- **Directional variation:** compare operator-first, audit-first, compact expert, or explanatory presentations.
- **Conceptual variation:** compare different report or command-flow models.

Use the cheapest level that supports a real decision. Broad interaction changes belong early; later work benefits from smaller variants around an accepted direction.

## Assign distinct candidates

Three candidates are a useful default. Possible assignments:

- **Operator-first:** foreground the immediate decision, prerequisites, and next permitted action.
- **Audit-first:** foreground identities, hashes, lifecycle evidence, retained artifacts, and exact failure causes.
- **Compact expert:** optimize for repeated use while keeping deeper evidence one clear step away.

Candidates should work independently during their first pass and use the same fixtures and fixed requirements.

## Require comparable evidence

Every candidate should use:

- identical manifests, resource indexes, and fixture installations;
- the same valid, invalid, stale, ambiguous, and partial cases;
- matching terminal widths or viewport sizes;
- the same build and test expectations;
- a branch and exact commit;
- complete captured output or matching screenshots;
- a short statement of decisions and compromises.

Useful checkpoints include:

- successful preparation;
- stale index or changed cache;
- ambiguous installation layout;
- missing permission;
- invalid comparison pair;
- texture-memory inspection;
- font generation and licensing guidance;
- narrow terminal output;
- recovery after a retained failed run.

## Review by decision

Compare individual decisions:

- Which version makes the next operator decision clearest?
- Which keeps claims close to supporting evidence?
- Which communicates invalid or incomplete comparison status honestly?
- Which makes mutation and restoration boundaries visible?
- Which remains usable during long, dense diagnostics?
- Which language could be misread under pressure?
- Which feels specific to Preflight rather than a generic CLI?
- Which ideas remain useful as more asset-quality tracks are added?

Agent review can check the brief, fixture coverage, omissions, consistency, and accessibility. Human review owns presentation and emphasis.

## Converge explicitly

Record accepted and rejected decisions directly:

> Use A’s operator summary, B’s evidence table, and C’s compact memory breakdown. Preserve the current failure vocabulary. Reject the collapsed cache-restoration warning.

Choose one canonical branch. Give the convergence pass the original brief, accepted elements, rejected directions and reasons, unresolved details, and verification requirements.

Return to one implementation when the preferred direction can be described clearly, new candidates mostly rearrange details, or remaining concerns can be written as specific edits.

## Fidelity ladder

Choose the lowest-cost artifact that supports the decision:

1. written command or report outline;
2. sample terminal output;
3. static HTML or screenshot;
4. isolated renderer;
5. working branch;
6. complete interactive flow.

Use working code when prompts, terminal wrapping, state transitions, lifecycle detection, or evidence navigation determine quality.

## Possible uses in Starsector Preflight

This workflow may be useful for comparison reports, readiness and lifecycle output, texture-memory inspection, manifest diagnostics, recovery guidance, operator permission prompts, font-generation reports, asset-quality documentation, and any future visual inspector.

Keep observed identities, hashes, cache controls, permission requirements, failure classifications, licensing rules, and retained evidence fixed across candidates. Explore how operators encounter and understand those facts.
