## Immediate implementation goal

The exact-gated live texture compatibility pilot completed its acceptance route through the game main menu and a normal exit on 2026-07-19. The remaining goals are repeatable OFF-versus-ENABLED measurement for the accepted compatibility path, then the same acceptance route with prepared pixels.

The first 2026-07-18 launcher-stage pilot proved one exact transformation and 20 of 20 cache hits. A follow-up reached the title-screen render path with 17,014 hits and 66 safe fallbacks, then failed in `LayeredRenderer.renderExcluding` with a null iterator. An adapter-OFF control with the same enabled-mod list reached the main screen and exited normally, with no adapter report, no texture cache context, and no bounded fatal-lifecycle match. The runs differ in generated image bytes and heap size, so they are not timing comparators, but the compatibility path remains implicated and disabled. The branch also adds bounded fatal-log lifecycle detection because the enabled launcher shell returned `0` after the fatal game-thread stack. See [the retained evidence](evidence/2026-07-18-real-texture-preparation-and-compatibility-pilot.md).

Source review after the control found that both version-1 designs bypassed `com.fs.graphics.L.class(String)`, Starsector's asynchronous image-preloader rendezvous. Compatibility-v2 executes that handoff first, lets a preloaded image win, and consults Preflight only on the original direct-decode branch. Packaged tests cover preloaded wins, warm hits after exactly one handoff, and one-time direct-decode fallback. A read-only transform check succeeds on the exact installed class for compatibility-v2, and the controlled version-2 acceptance run passed on 2026-07-19. Activation remains explicit opt-in per run.

The same check declines prepared-pixels-v2 at the older color-sink matcher. The installed converter stores derived colors on `TextureLoader` and later invokes texture-object setters; the synthetic fixture modeled direct texture-object fields. Do not run or claim the prepared-pixel path until that exact dataflow and fixture are corrected and the installed-byte check passes.

The pilot should reuse a verified prepared texture artifact at the narrowest reviewed image seam while preserving the remainder of Starsector's original texture and preload path.

### Required behavior

1. The target activates only when all manually reviewed identity fields match:
   - class SHA-256;
   - required method descriptors;
   - source archive SHA-256 and suffix;
   - source kind;
   - loader class and loader name.
2. Unknown or changed installations remain untouched.
3. A valid cache hit returns semantically equivalent data.
4. Any miss, stale record, checksum failure, unsupported image, ambiguity, internal exception, or circuit-breaker state calls the original method.
5. The game installation, launcher, mods, saves, and VM parameter files are never edited.
6. The existing environment/property kill switch remains authoritative.
7. The adapter emits bounded hit, miss, fallback, corruption, and disable-reason counters.

## Benchmark protocol for the first live target

Use a stable updated-mod profile that can complete the scenario and exit normally.

For each mode, collect at least five successful runs:

1. OFF cold-ish;
2. OFF warm;
3. ENABLED build/miss;
4. ENABLED warm hit;
5. ENABLED corrupt artifact;
6. ENABLED after changed source or mod order.

Report every run plus median/min/max. Do not delete failed runs; classify their cause.

Minimum metrics:

- main-menu and campaign-readiness time;
- texture hits, misses, fallback reasons, and bytes served;
- original image-decode and conversion work remaining in JFR;
- peak heap and direct-buffer use;
- cache bytes and entry count;
- normal exit status.
